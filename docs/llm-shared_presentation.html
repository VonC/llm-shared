1. Problème
Le problème du « vibe-coding »
Générer du code vite sans spécifier ce qu'on veut vraiment
💡
L'idée
« J'ai une idée, génère-moi du code, je verrai les détails plus tard. »
⚡
Le code arrive vite
Mais il s'appuie sur des hypothèses non écrites. Il casse au premier cas réel non prévu.
💥
La conséquence
Code sans documentation d'intention. Difficile à maintenir, à expliquer, à étendre — même par l'auteur.

⚠️ Ce que refuse le workflow

Passer directement à l'implémentation sans exigence écrite · Sans design · Sans tests · Sans scénarios d'acceptation

6. Implémentation
Phase d'implémentation — chaîne automatisée
pw handoff enchaîne les étapes sans menu ni « go-ahead » entre les cases
ZONE AUTOMATISÉE — un seul /implement-step N déclenche la chaîne
/implement-step N
Code + tests · ghog day vert
▼
/implementation-check N
Verdict Oui/Non
↻ Non → /implement-missing-step
Combler les écarts, ghog day, re-check
ou
✓ Oui → /group-commits-msg
Messages de commit groupés → a.commit

🧪 Tests seuils d'abord

Le plan commence par des tests qui échouent volontairement. Les étapes suivantes les font passer un par un.

📦 Commits groupés

L'IA lit le diff, groupe les fichiers du moins dépendant au plus dépendant, écrit un message par groupe.

🛑 Portail de commit

La chaîne s'arrête à a.commit. L'auteur révise, dit « go ahead », puis gcba rejoue les commits.

7. Commits & Release
Commits conventionnels & préparation de release
Au-delà du changelog : le « pourquoi » pour le prochain lecteur
Format de commit étendu
feat(scope): sujet (52 car. max)

Why:
Raison du changement (paragraphe 1).
État « après » du code (paragraphe 2).

What:
- modification 1
- modification 2
- modification 3
Le titre alimente le changelog. Le corps donne le contexte que le diff ne porte pas — pour l'humain et pour l'IA qui relira plus tard.

8. Groundhog
Groundhog — la boucle de tests
Un seul outil remplace les anciens alias : ghog day marche la chaîne et s'arrête au premier problème
ghog day
▼
1. check.bat
Compilation + lint
→
2. Tests affectés
Rapide, sans couverture
→
3. Suite complète
Couverture 100% exigée
▼
✓ Objectif atteint : tests verts + couverture au seuil

🔄 Boucle de correction IA

L'IA lit le code de sortie, applique le correctif nommé dans le rapport, relance ghog day. Arrêt si aucun progrès.

📊 Sortie maîtrisée

Petite pour l'IA (logs redirigés, tail lu). Vivante pour l'humain (barre de progression, rapport final).

🆔 État de cycle

Contrat fichier a.ghog.status. L'IA ne devine jamais : ghog status lit l'état, exit 6 = en cours, 7 = tué.

Bénéfices
Bénéfices pour la DSI
Ce que le framework apporte aux équipes et à la qualité

📋 Traçabilité complète

Chaque décision est documentée. Le « pourquoi » est explicite, pas à déduire du code. Revu par l'IA avant validation.

🤖 Multi-agent

GitHub Copilot, Claude Code, ChatGPT Codex. Les mêmes instructions markdown, les mêmes artefacts.

🧪 Qualité par construction

Tests seuils avant implémentation. Couverture 100% exigée. Revue d'implémentation systématique par l'IA.

📚 Documentation vivante

Brouillon, exigence, design, plan, validation, commits — chaque phase laisse un artefact relu et consolidé.

⚡ Automatisation

pw skill et pw handoff enchaînent les étapes sans intervention. L'humain intervient seulement aux points de revue.

🔓 Licence MIT

Réutilisable dans tous les projets. Copie, fork, adaptation sans contrainte de copyleft.
