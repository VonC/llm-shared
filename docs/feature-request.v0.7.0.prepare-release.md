# Release process

## Current process

Right now, when I want to make a release, I do the following:

- make sure the status is clean and that every step of the current plan are validated and committed
- make sure you identify a development effort since the last tagged release, and that there is at least one commit with feature/issue/design/plan changes since the last tagged release. If not, return a message saying that there is nothing to release yet, because no development effort is in progress. Tht means identifying the target version and the slug of the release (which if not developed on main, should be the name of the branch you are in), and checking that there is at least one commit with feature/issue/design/plan changes since the last tagged release. If not, return a message saying that there is nothing to release yet, because no development effort is in progress.
- switch back to main, ignoring other worktree (I do not meant just using --ignore-other-worktrees, but also actually ignoring other folder like "'myproject_main", which is a worktree of the same repo, but not the main one, and which might become unsynced with main: ignore that, I will deal with it later)
- git merge --no-ff the branch of the current plan, which is now fully validated and committed

If I was developing on main directly, only the first step is needed, and the merge is not needed, since main is already up to date. If status is not clean, I propose to add and "group commit messages" first, to make sure everything is committed before continuing with the release process.

Then I run the `update-merge-commit-msg` skill to update the merge commit message with a conventional commit message that includes the version number.

The I check version.txt: if the version number still reflect the old version, I update it to the new version number -SNAPSHOT. Note; the version is reflected in the name of the design or plan document, so you can derive it from there.

Then I run the `prepare-release-notes` skill to prepare the release notes. This skill will update version.txt content.

If there is a `pyproject.toml` file at the root folder of the project from which this process is done, I update the version number (no -SNAPSHOT needed there) in pyproject.toml to the new version number. And I run `uv sync` and check that the version number is reflected in the generated `uv.lock` file.

Finally, I run "update-changelog.bat" script to update the changelog file with the new version number and the release notes.

Then I commit all those changes as "chore(release): prepare for vX.Y.Z release".

And I manually review everything before doing a brel (build release) which creates the tag.

## Goal: automate the release process

Write a skill that automates the release process, including the steps above, and that can be run from any branch (not just main).

Make sure the skill is able to call other skills like `update-merge-commit-msg`, `prepare-release-notes`, and that those skills have a handoff reminding them that, if they are called from the release skill, they should then proceed with the release skill once they have done their job, meaning return to the release skill to continue the process. But if they are called directly, they should not return to the release skill, but just finish their own job and return to the caller.

Propose 3 different name for the skill, and explain why you chose each name. Once the name is chosen, write the skill, and make sure it is well documented, with a clear description of what it does, and how to use it. Make sure the skill is idempotent, meaning that if it is run multiple times, it will not cause any issues or errors. If run on a release that just been done, it should not do anything, and should just return a message saying that the release is already done. If run when there is no commit with feature/issue/design/plan changes, it should return a message saying that there is nothing to release yet, because no development effort is is progress.

## Review first, then implement

Follow the /review-ask-questions and /consolidate-then-review-ask-questions loop on this very document `docs/feature.v0.7.0.prepare-release.md` (originally drafted as `a.release-skill.md`) to make sure the skill is well designed and that it meets the requirements before implementing it. Make sure to ask questions about any unclear requirements or edge cases, and consolidate the answers into a clear design before proceeding with the implementation. Once you have no more questions, and the design is clear, proceed with the implementation of the skill as specified in the Goal section above. Make sure to test the skill thoroughly before considering it done, and document any known limitations or issues.

## Agreed behaviour for the prepare-release skill

The nine open questions are answered. The skill must behave as follows:

1. Skill name (Q1): the skill is named `prepare-release`. It readies every release artifact and stops before the tag-cutting brel, which a human runs.

2. Trigger and the nothing-to-release case (Q3): the last released tag is read with `git describe --tags --abbrev=0`. A development effort exists only when at least one commit since that tag adds or changes a `docs/{feature,issue,design,plan}.vX.Y.Z.*.md` file. With no such commit, the skill prints "nothing to release yet, no development effort in progress" and exits.

3. Already-released guard (Q7): when HEAD sits exactly on a tag (`git describe --tags --exact-match` succeeds) and no effort commit (the rule from item 2) follows that tag, the skill prints "release already done" and exits without touching the tree. This makes repeated runs safe.

4. Target version and slug (Q4): the version comes from an effort document touched since the last tag, matching `docs/{feature,issue,design,plan}.vX.Y.Z.*.md`, preferring a design or plan document when one exists and falling back to the feature-request or issue document for a design-free effort (this very release is one such case, it has only a feature-request). The slug comes from that document topic, or from the branch name when the work is off main. The version is cross-checked against the `X.Y.Z-SNAPSHOT` line in version.txt. No candidate means nothing to release; several different versions makes the skill stop and ask.

5. Dirty working tree (Q8): a dirty tree stops the skill. It lists the dirty files and offers to run group-commits-msg, continuing only once the tree is committed clean.

6. Branch and worktree handling (Q9): the skill switches the current worktree to main with `git switch --ignore-other-worktrees main` and runs `git merge --no-ff` of the effort branch there. The `--ignore-other-worktrees` flag lets the switch proceed in place even when main is also checked out in a sibling worktree such as `..._main`; the skill never reads, switches, syncs, or checks the state of that sibling, which may end up out of sync with the new main tip (the user deals with it later). When HEAD is already main, it skips the merge. Only the current worktree's cleanliness matters (item 5); the skill never stops just because main is checked out elsewhere.

7. Sub-skill handoff (Q5): the skill uses a flag file `a.prepare-release.active` at the project root (git-ignored by the `a.*` rule). It deletes any stale flag file at the start of the run, creates the flag file before each sub-skill call (`group-commits-msg`, `update-merge-commit-msg`, `prepare_release_notes`), and removes it on every exit. Each sub-skill checks for the file: when it exists, the sub-skill finishes its job and returns control to the release skill; when it is absent (a direct call), it finishes standalone, including its own closing message. A flag file works across separate shells, where an environment variable would not survive between commands, and the delete-at-start makes a stale flag self-healing. The merge-commit-msg step is skipped when there was no merge (the on-main case of item 6).

8. Release artifacts and the single prepare commit (Q6): the skill writes the `X.Y.Z-SNAPSHOT` line in version.txt and, when a pyproject.toml exists, the release `X.Y.Z` there, then runs `uv sync` and checks uv.lock. It delegates the version.txt release-notes summary and the CHANGELOG.md update to `prepare_release_notes`, and does not run update-changelog.bat as a redundant extra step right after it (the only later re-runs are the Step 10 review-pause regenerations that apply the user's edits, see item 12). It records everything, version.txt, CHANGELOG.md, the `.changelog.fixes` when the review pause changed it, and pyproject.toml and uv.lock when present, in one `chore(release): prepare for vX.Y.Z release` commit. pyproject.toml sits at the release `X.Y.Z` while version.txt stays `-SNAPSHOT` until brel; that gap is intended.

9. Stop boundary (Q2): the skill stops at the prepare commit, prints a summary of what changed, and tells the user to review and run brel. It never creates a tag.

10. Base the branch on the latest main before merging (Q10): before the item 6 switch and merge, and only off main, the skill fetches `origin/main` (read-only) and first brings local main up to date. When local main is strictly behind `origin/main`, it resets the local main ref to `origin/main` with `git update-ref` (it ignores the sibling worktree that holds main, which then falls out of sync); when local main and `origin/main` have diverged, it stops and asks rather than drop local commits. It then checks whether local main is an ancestor of the branch tip with `git merge-base --is-ancestor main HEAD`. When it is, the merge proceeds as in item 6. When it is not, the skill stops and offers three choices: (a) rebase the branch onto main, then run a `ghog day` green gate, then merge; (b) merge `--no-ff` anyway, then run the `ghog day` green gate after the merge (only in this stale-base case); or (c) abort, removing the flag file and changing nothing. The green gate runs the `ghog day` groundhog loop to a green suite; when fixes were needed, it commits them through `group-commits-msg` with the user's review and go-ahead before the release continues. A rebase conflict stops for the user to resolve, then resumes non-interactively with `cmd /V /C "set "GIT_EDITOR=true" && git rebase --continue"`. With no remote, the latest-main reference is the local `main`.

11. Merge commit message structure (Q11): the merge commit message from item 6 must carry the group-commits-msg `Why:` / `What:` structure, a subject line, a `Why:` section (the reason and the now-state it allows), then a `What:` section with a dashed change list. It must be produced by the `update-merge-commit-msg` skill, which fills that template from the merged documents and enforces the shape; the reword applies it. The release skill must not skip the reword (Step 7) and must never hand-type the message with a free-form `git commit --amend -m`, which yields a bare subject and one paragraph with no `Why:` / `What:`. After the reword, the skill verifies the merge body has both sections, and re-runs the reword when it does not, before continuing.

12. Review pause for version.txt and the changelog (Q12): after Step 9 writes version.txt and CHANGELOG.md, and before the pyproject step, the skill stages version.txt and CHANGELOG.md as a baseline and pauses. The user may edit the version.txt release-note summary and request the creation or update of `.changelog.fixes` (the changelog substitution rules); the pause is for release-notes content only, not for coding a project fix. The skill resumes only on the user's "go ahead". On go-ahead it checks `git diff` on version.txt and CHANGELOG.md (and whether `.changelog.fixes` was touched); when anything changed it regenerates CHANGELOG.md with update-changelog, re-stages, and pauses again, looping until a "go ahead" follows no further edit. A `.changelog.fixes` change is committed in the single prepare commit (item 8).

Two scope points are carried as defaults (raise either now if it is wrong):

- Remote: the whole process is local. The skill does not push to a remote; pushing stays a manual step after brel.
- Project reach: the skill runs from any project that has a version.txt and the dev_workflow build (brel), llm-shared being one such project. The pyproject.toml and uv step runs only when a pyproject.toml exists.
