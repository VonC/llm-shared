# git_history_dashboard

A standalone, single-file HTML dashboard of a project's commit history:
weekly activity stacked by conventional commit type, a GitHub-style daily
heatmap, weekday and hour-of-day distributions, top scopes, and the most
recent commits.

This tool is **shared**: it lives in `llm-shared` but always builds the
dashboard for the **calling project**. It resolves that project's root
through the shared `find_project_root` helper, which prefers the `PRJ_DIR`
environment variable each project's `senv.bat` exports — the same
mechanism `git_batch_commit` uses. So it can be run from any project, from
any working directory.

## Files

| File | Role |
| --- | --- |
| `build.py` | Exports the calling project's commit history with `git log`, then renders `data.json` + `dashboard.html`. |
| `template.html` | HTML template with placeholders (`__DATA__`, `__TOTAL_COMMITS__`, …) that `build.py` substitutes. |
| `__init__.py` | Makes the folder an importable package for the test suite. |

## Where the output goes

The build writes into the **calling project**, under
`<project>/docs/git_history_dashboard/`:

| Output | Versioned? | Why |
| --- | --- | --- |
| `dashboard.html` | Yes | The rendered, self-contained dashboard — committed to record the project's history snapshots. |
| `data.json` | Yes | Aggregated commit data — committed alongside the dashboard. |
| `git_history.csv` | No (git-ignored) | Pipe-separated `git log` export; a scratch artifact. |

## Run it

The simplest way is the `ghd` alias (defined in `llm-shared/senv.doskey`,
loaded by every project's `senv.bat`). It refreshes the history export,
rebuilds the dashboard, and opens it in the browser — in one step:

```text
ghd
```

Or call `build.py` directly. With no arguments it runs the documented
one-liner —

```text
git log --all --pretty=format:'%H|%ai|%an|%s' --date-order > git_history.csv
```

— writing `git_history.csv` into `<project>/docs/git_history_dashboard/`,
then parses that file:

```text
python <llm-shared>/tools/git_history_dashboard/build.py
```

Against a different working copy:

```text
python build.py --git-dir /path/to/repo
```

From a pre-exported CSV (skips the `git log` call):

```text
python build.py --csv git_history.csv
```

Pass `--out-dir` to redirect the dashboard, `--template` to use a
different HTML template.

## What the dashboard shows

- **Header metrics** — total commits, active-day ratio, peak day, peak week.
- **Weekly commit activity** — stacked bars per ISO week (Monday-anchored),
  one stack per commit type. The chips above the chart toggle individual
  series on/off; hovering a bar shows the per-type breakdown for that week.
- **Daily activity** — a GitHub-style heatmap covering every day in the
  range, with five intensity buckets (0 / 1–3 / 4–7 / 8–15 / 16+).
- **By weekday / By hour of day** — twin distribution panels.
- **Top focus areas** — the 15 most-used conventional-commit scopes.
- **Most recent commits** — last 10 entries, type-coloured.

## Observations block

`template.html` carries a hand-written `<section class="observations">`
block with commentary on the activity shape. It is **not** regenerated by
`build.py` — when a project's data changes meaningfully, edit that block
in the template and re-run the build.

## Dependencies

- **Build time:** Python 3.13 standard library only, plus the shared
  `tools.find_project_root` helper. No third-party packages.
- **View time:** any modern browser. Chart.js 4.4.1 is loaded from
  `cdnjs.cloudflare.com` at view time; everything else (data, layout,
  calendar heatmap) is inlined into `dashboard.html`.

## How the data is parsed

Each commit subject is matched against the conventional-commit pattern
`<type>(<scope>)!: <subject>` to extract its type and scope. Unrecognised
types collapse into `other`. The aggregation produces:

- a contiguous daily series (zero-filled), and a Monday-anchored weekly
  rollup keyed by ISO week start;
- per-type totals across the whole history and per day/week;
- top-15 scopes ranked by frequency;
- 24 hour-of-day buckets and 7 weekday buckets (Mon=0);
- the 10 most recent commits as `{sha, date, type, scope, msg}`.

If you change the type or scope conventions, edit `KNOWN_TYPES` and the
regex constants at the top of `build.py`.
